# TASK 관리 — 그린 책임

> 모든 업무는 **`docs/프로젝트/TASK.md`** 단일 파일에서 통합 관리한다.
> TASK.md는 "나중에 정리"가 아니라 **작업 흐름 속에서 실시간으로** 갱신한다.

## TASK.md 구조

- **테이블 형식**: `| # | 작업 | 중요도 | 담당 | 상태 | 비고 |`
- **중요도**: P1(긴급) > P2(중요) > P3(향후)
- **상태 흐름**: `예정` → `요청` → `진행` → `완료`
- 완료 시 작업 현황에서 삭제 → "완료" 섹션 최상단에 이동
- **TODO list**: 향후 검토 사항은 TASK.md 하단 TODO 섹션에서 관리
- 개별 문서에는 TASK를 두지 않고 TASK.md 링크만 유지

## 그린의 역할

- 회의/대화에서 나온 **모든 task를 즉시** TASK.md에 등록 (담당자 + 중요도 배정)
- 세션 시작 시 TASK 현황 점검, 지연 항목 보고
- 새로운 요청·버그·개선 포인트 발견 시 대화 중이라도 즉시 등록

## 실시간 갱신 규칙 (필수)

| 트리거 | 행동 |
|--------|------|
| **작업 착수 시** | "현재 진행 중" 테이블에 등록 + 작업 현황 상태 → `진행` |
| **작업 완료 시** | 작업 현황에서 삭제 → 완료 섹션 최상단에 추가 + "현재 진행 중" 비우기 |
| **새 작업 발견 시** | 즉시 작업 현황에 등록 (대화 중 나온 요청, 버그, 개선 포인트 등) |
| **커밋 직전** | TASK.md 상태가 실제와 일치하는지 점검, 불일치 시 갱신 후 커밋 |
| **세션 종료 시** | 미완료 작업은 "현재 진행 중"에 진행 상황 기록, 완료된 건 정리 |

> 완료된 작업이 작업 현황에 남아있으면 안 되고, 진행 중인 작업이 기록 없이 진행되면 안 된다.

## TASK 상세 로그 — `docs/프로젝트/task/{TASK_ID}.md`

모든 TASK에 쓸 필요 없음. **아래 경우에만** 작성:

- 설계 결정이 포함된 작업 (왜 A를 선택하고 B를 기각했는지)
- 여러 파일을 변경한 작업 (변경 추적 필요)
- 나중에 "왜 이렇게 했지?" 질문이 예상되는 작업
- 사용자 검토가 필요한 작업 (미결 이슈 기록)

**작성 시점**: 작업 완료 시 또는 중요 결정이 내려진 시점

**템플릿**: `docs/프로젝트/task/TEMPLATE.md` 참조

**필수 섹션**: 배경, 결정 사항(결정+이유+대안), 변경 내역, 검증, 미결 이슈
