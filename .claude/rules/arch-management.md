# 아키텍처 관리 — Architecture Decision Records

> 코드는 지금 상태를 보여주지만, **왜 그렇게 됐는지**는 설명하지 않는다.
> 아키텍처 결정은 `docs/arch/` 에 ADR(Architecture Decision Record)로 기록한다.

## ADR vs TASK 상세 로그 구분

| 문서 종류 | 저장 위치 | 목적 | 수명 |
|-----------|-----------|------|------|
| TASK 상세 로그 | `docs/프로젝트/task/` | 특정 작업의 실행 기록 (버그 수정, 구현 과정) | 해당 작업 완료 후 참고용 |
| **ADR** | `docs/arch/` | 시스템 구조 결정의 영구 기록 | 구조가 바뀔 때까지 유효 |

같은 작업도 두 문서가 공존할 수 있다.
예: C54(TF 부모 충돌 수정) → TASK 상세 로그에 버그 수정 과정 기록 + ARCH-001에 TF 트리 구조 업데이트

## ADR 작성 기준 (이진 체크리스트)

아래 항목 중 **하나라도 해당**하면 ADR을 작성(또는 기존 ADR 업데이트)한다.

| # | 조건 | 예시 |
|---|------|------|
| A1 | **시스템 레이어 경계** 결정 | "TF는 어느 노드가 발행하는가" |
| A2 | **외부 라이브러리/프레임워크 선택** | SmacPlannerLattice vs TEB, MPPI vs DWA |
| A3 | **데이터 흐름 구조** 변경 | 새 센서 추가, 토픽 리매핑, 네임스페이스 변경 |
| A4 | **성능-안전성 트레이드오프** 결정 | costmap 해상도, EKF 주파수, 타임아웃 값 |
| A5 | 기존 ADR을 **폐기·대체**하는 결정 | AMCL → static TF로 교체 |

## ADR 파일 구조

```
docs/arch/
├── TEMPLATE.md
├── ARCH-001_tf_tree.md
├── ARCH-002_dual_ekf.md
├── ARCH-003_nav2_stack.md
└── ...
```

### 파일명 규칙

`ARCH-{NNN}_{짧은_제목}.md` — 번호는 생성 순서, 변경하지 않는다.

### 상태값

- `채택됨` — 현재 시스템에 적용 중
- `폐기됨` — 더 이상 사용하지 않음 (이유 기록)
- `대체됨 → ARCH-NNN` — 다른 ADR로 대체됨 (링크 명시)

---

## ADR 작성/갱신 시점

| 트리거 | 행동 |
|--------|------|
| **새 구조 결정** | 새 ADR 작성 (ARCH-NNN) |
| **기존 구조 변경** | 해당 ADR의 상태를 `대체됨` → 새 ADR 작성 |
| **구조 제거** | 해당 ADR의 상태를 `폐기됨`으로 변경 + 이유 기록 |
| **TASK 완료 시** | TASK에서 아키텍처 영향이 있으면 관련 ADR 업데이트 |

---

## 그린의 역할

- 아키텍처 영향이 있는 TASK 완료 후 ADR 작성 또는 업데이트
- 세션 시작 시 `docs/arch/` 인덱스 확인 (대규모 작업 전)
- 새 구조를 논의할 때 기존 ADR 참조하여 일관성 유지
