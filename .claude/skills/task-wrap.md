# Task 정리 스킬 (task-wrap)

> **트리거**: 아래 표현이 나오면 이 스킬을 즉시 실행한다.
> - "task 정리해줘" / "task 정리"
> - "지금까지 정리해줘" / "지금까지 사항 저장"
> - "session 마무리" / "마무리해줘"
> - "task 브리핑" / "브리핑해줘"
> - "문서화해줘"

---

## 실행 순서 (순서 준수 필수)

### STEP 1 — 현재 상태 스캔

다음 파일을 읽는다:
- `docs/프로젝트/TASK.md` — 진행 중, 완료, 미결 현황

### STEP 2 — TASK.md 갱신

아래 순서로 TASK.md를 업데이트한다:

1. **🔥 현재 진행 중** — 이번 세션에서 달라진 진행 상황과 다음 할 일 최신화
2. **작업 현황** — 상태 변경 (예정→진행, 진행→완료)
3. **완료** — 이번 세션에서 완료된 태스크를 완료 섹션 **최상단**에 추가
4. **날짜** — 상단 `> 마지막 갱신:` 갱신

### STEP 3 — 문서화 판단 (T1~T5 체크)

완료된 각 태스크에 대해 `task-management.md`의 기준으로 판단한다:

```
T1 수정 파일 3개+  →  task/{ID}.md 작성
T2 대안 비교 포함  →  task/{ID}.md 작성
T3 미결 이슈 존재  →  task/{ID}.md 작성
T4 P1 완료         →  task/{ID}.md 작성
T5 아키텍처 영향   →  docs/arch/ 우선 확인 (아래 STEP 4)
```

해당 없으면 → TASK.md 한 줄 요약으로 충분, 문서 생략

### STEP 4 — ADR 판단 (A1~A5 체크)

완료된 태스크 중 아키텍처 영향이 있는 것에 대해 `arch-management.md`의 기준으로 판단한다:

```
A1 시스템 레이어 경계 결정  →  새 ARCH 작성 또는 기존 ARCH 업데이트
A2 라이브러리/프레임워크 선택  →  새 ARCH 작성
A3 데이터 흐름 구조 변경  →  기존 ARCH 업데이트
A4 성능-안전 트레이드오프  →  새 ARCH 또는 기존 업데이트
A5 기존 결정 대체  →  기존 ARCH 상태를 '대체됨' 처리 + 새 ARCH 작성
```

작성/수정 후 `docs/arch/INDEX.md` 업데이트

### STEP 5 — 빌드 필요 여부 확인

코드/설정 파일이 변경됐으면:
```bash
colcon build --packages-select ad_bringup [기타 변경된 패키지]
```
빌드 성공 여부를 TASK.md 또는 task 로그에 기록

### STEP 6 — 브리핑 출력

다음 형식으로 현황을 요약해 사용자에게 전달한다:

---

**📋 Task 브리핑 — {날짜}**

**🔥 진행 중**
- `{TASK_ID}` {작업명}: {현재 상황} → {다음 할 일}

**⏸️ 사용자 액션 대기**
- `{TASK_ID}` {남은 사용자 액션}

**✅ 이번 세션 완료**
- `{TASK_ID}` {작업명}
- 문서: `{작성된 파일 목록}`

**📌 다음 우선 작업**
1. {P1 또는 블로커 제거 작업}
2. {그 다음}

---

## 판단 규칙 요약

| 상황 | 행동 |
|------|------|
| 단순 버그 수정 1~2파일 | TASK.md 한 줄만 |
| P1 완료 | task/{ID}.md 필수 |
| 파일 3개+ 수정 | task/{ID}.md 필수 |
| TF·센서퓨전·Nav2 구조 변경 | docs/arch/ 업데이트 |
| 새 기술 스택 선택 | docs/arch/ 신규 작성 |
| 이미 문서 있는 태스크 | 기존 파일 업데이트 (중복 작성 금지) |
| 링크 있는데 파일 없음 | 링크 제거 또는 인라인 요약으로 대체 |
